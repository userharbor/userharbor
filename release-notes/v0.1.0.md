## ⚡ v0.1.0 - FastAPI Adapter

This release introduces the first public FastAPI adapter for UserHarbor, giving
applications ready-to-use authentication routes and dependency helpers while
keeping account-management logic in the framework-agnostic core package.

## ✨ Added

* ⚡ **FastAPI router**
  * Added `UserHarborFastAPI(harbor).router`
  * Added built-in account routes for registration, email verification, login,
    logout, password reset, password change, and account deletion
  * Added `GET /me` for returning the current authenticated user
  * Routes delegate account behavior to the configured `UserHarbor` instance

* 🔐 **Bearer session authentication**
  * Added bearer token extraction from the `Authorization` header
  * `POST /login` returns a bearer-compatible token response with
    `access_token` and `token_type`
  * Protected routes use UserHarbor session tokens directly

* 🧩 **FastAPI dependency helpers**
  * Added `auth.current_user` for required authenticated users
  * Added `auth.optional_user` for routes that can work with or without a
    session
  * Added `auth.require_role(role)` for role-protected routes
  * Added `auth.require_permission(permission)` for permission-protected routes
  * Added public `bearer_token` helper for applications that need direct access
    to the raw session token

* 👤 **User serialization**
  * Added default serialization for UserHarbor user objects with `username`,
    `email`, and `verified`
  * Added configurable `user_serializer` support for applications that return
    custom public user shapes

* 📦 **Request and response schemas**
  * Added Pydantic request models for registration, login, verification,
    password reset, password change, and account deletion
  * Added `TokenResponse` for login responses
  * Added `MessageResponse` for neutral account operation responses

## 🔒 Security

* 🚫 **HTTP error mapping**
  * UserHarbor authentication failures are mapped to `401 Unauthorized`
  * authorization failures are mapped to `403 Forbidden`
  * unknown roles and permissions are mapped to `404 Not Found`
  * invalid input and token failures are mapped to `400 Bad Request`
  * username conflicts are mapped to `409 Conflict`

* 🧾 **Structured error responses**
  * Added stable error payloads with `detail` and `code`
  * UserHarbor exception types are preserved as machine-readable error codes
  * Session authentication failures remain distinct from role and permission
    authorization failures

## 📖 Documentation

* Added README usage example with `FastAPI`, `UserHarbor`, and
  `UserHarborFastAPI`
* Documented built-in authentication routes
* Documented bearer token usage through the `Authorization` header
* Documented dependency usage for current users, roles, and permissions

## ⚠️ Compatibility Note

This is the first public release of `userharbor-fastapi`.

Applications should provide a configured `UserHarbor` instance with a concrete
`UserStore` and `EmailSender`. The adapter does not include database storage or
email delivery; those remain separate UserHarbor integrations.

The adapter uses the synchronous UserHarbor core API. FastAPI will run standard
sync route handlers in its normal execution model, which fits the current
UserHarbor storage and email contracts.

## ⚠️ Project Status

UserHarbor and its integrations are still in an early stage of development and
are not yet recommended for production use. This release focuses on providing a
small, predictable FastAPI integration layer without moving framework-specific
behavior into the core package.
